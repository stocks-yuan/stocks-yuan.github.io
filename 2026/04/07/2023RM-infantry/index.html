二、多任务调度与通信

2.1 自定义调度器

自定义的 scheduler 调度器本质上是 非抢占式时间片轮询调度器，通过硬件定时器来管理各个任务按时运行。

  • 优点：代码极其轻量，不需要像 FreeRTOS 一样消耗内存开辟任务栈，所有任务类通过new方法在堆中分配内存。由于完全基于微秒硬件定时器运行，对于 Robomaster 机器人的姿态解算、外环内环 PID 运算这种要求 执行频率非常稳定 的系统来说，由于能把 dT_us 精确传入算法，因此控制非常精准和平滑。

  • 缺点/限制（协作式的妥协）：它是非抢占式的。如果 app 层的某个任务偷偷加了一句 HAL_Delay(5) 的死延时阻塞代码，由于中断不能被切换，这将会导致整个数组里的其他任务都卡住不能执行。因此，编写应用业务的程序员严禁在 update() 函数中使用任何阻塞式的等待代码。

核心数据结构与抽象

在 Scheduler 目录下，调度器主要由两个核心类Task_Base和Scheduler构成：

  • Task_Base（任务基类）：系统里所有的任务（如云台任务 GimbalControlTask ）都必须继承自这个类。它定义了三个纯虚函数供子类去实现具体业务代码：

    • init()：任务首次执行时的初始化。

    • update(timeus_t dT_us)：任务需要循环执行的核心逻辑。

    • uninit()：任务被注销、销毁时执行的清理逻辑。

    • 包含时间属性：interval_tick_us（期望的调度周期，即每隔多少微秒跑一次）、last_tick_us（上次执行时间）等。

  • Scheduler（调度器类）：是整个系统任务的大管家。内部持有一个任务指针数组 tasks[MAX_TASK_NUM] 和一个异步函数数组 asyncs，负责统一管理和调度运转。

调度运行详细工作流程

自定义调度器的工作流程集中体现在 Scheduler::run(void) 函数中，这个函数会被放在主程序的无尽循环中高频不停地调用。

调度过程如下：

第一步：处理异步延时任务（AsyncFunc）

  • 调度器首先检查异步函数数组。如果某个动作只是想“延迟 X 毫秒后执行一次”（如闪烁一次 LED，或者延迟闭合某个继电器），调度器通过比对时间戳判断是否达到了延迟时间。一旦时间到，触发调用并标记为已执行，不阻塞主循环。

第二步：遍历所有注册轮询的任务（Tasks）

  • 调度器会遍历 tasks 数组中的各个常规任务。

第三步：精确的时间片比较（调度核心）

  • 这是一种基于时间差计算的轮询：获取当前系统微秒时间 time_now_us（通过底层读取定时器计数得到的大数）。

  • 计算当前时间与该任务上一次执行的时间记录（last_tick_us）之间的差值 delta_time_us。

  • 触发条件：如果 delta_time_us >= tasks[i]->interval_tick_us（即经过的时间已经大于等于该任务设定的执行周期），则认为该任务“时间到了，需要执行”。

第四步：状态机流转与执行（生命周期）

如果一个任务可以被执行了，调度器会做如下处理：

  1. 更新时间戳：立马把 last_tick_us 设置为当前时间。

  2. 生命周期检查：

    • 检查 inited 标志位。如果是首次执行（未初始化），则先调用任务的 init() 方法，并将 inited 置为 true。

    • 如果已初始化过，则调用核心的 update(delta_time_us)，并将真实的时间差（dT）传给内部。（在此项目中这个 dT 非常关键，它会被用于 PID 运算时的积分项和微分项的时间参数，使算法不受任务微弱波动的影响）。

  3. 自杀机制（Self-Kill）： 如果任务在内部逻辑调用了 self_kill()，调度器在本次执行完之后，会调用 unregisterTask 处理后事，执行任务的 uninit()，然后把它从调度队列里除名踢出。

  4. 统计运算： 记录任务执行的结束时间戳，用以统计单次任务到底占用了多少 CPU 时间 real_interval_tick_us，在 debug 模式下还会顺便把执行频率算出来。

2.2 DMA双缓冲区+串口空闲中断

使用 双缓冲区 (Double Buffer) + DMA (Direct Memory Access) + 串口空闲中断 (IDLE Interrupt) 来接收遥控器（如大疆的 DBUS 协议或富斯 IBUS 协议）数据，是目前行业内公认的最稳定、最高效的串口数据接收方案。

核心组件与原理

  1. DMA 原理： 当串口（USART3）接收到数据寄存器（USART3->DR）有新字节时，通常会触发 CPU 接收中断。但在 DMA 模式下，串口会向 DMA 控制器发送请求（USART_CR3_DMAR 被置位）。DMA 控制器接管总线，不需要 CPU 参与，自动将串口寄存器里的字节搬运到我们预先定义好的内存数组中（如 sbus_rx_buf）。 代码体现： hdma_usart3_rx.Instance->PAR = (uint32_t) & (USART3->DR); (设定外设源地址) 和 SET_BIT(huart3.Instance->CR3, USART_CR3_DMAR); (开启串口 DMA 接收)。

  2. DBM 双缓冲区模式 (Double Buffer Mode)

项目中开启了此模式，但是实际代码是在空闲中断中手动调整DMA指向的缓冲区

原理： 这是 STM32F4/F7 DMA 特有的一项硬件功能（DMA_SxCR_DBM）。顾名思义，我们给 DMA 提供两个内存块：Buffer 0 和 Buffer 1。

  • 当 DMA 开启时，它首先将数据写进 Buffer 0。

  • 当 Buffer 0 写满，DMA 硬件会自动、无缝地瞬间切换目标地址，开始向 Buffer 1 写入数据，同时触发一个中断。

  • 当 Buffer 1 写满后，它又能立刻切回 Buffer 0，形成乒乓操作。

  1. USART 空闲中断 (IDLE Interrupt) 原理： 一帧遥控器数据（例如 DBUS 是 18 个字节）发完后，总线会空闲一小段时间（没有后续数据）。STM32 的串口硬件能检测到这条线闲下来了，就会产生一个空余中断（IDLE 中断）。 作用： 我们可以利用这个中断来判断：“一整帧的数据到底接没接收完？” 如果触发了空闲中断，说明一帧数据已经完整接完了，此时我们在这中断里去处理刚满的 Buffer，并对 DMA 的接收剩余长度复位，防止发生错位。
执行流程
  1. 硬件准备阶段 在最开始的 RCProtocol::init() 中：
  • 程序为 DMA 指定了两个内存区：sbus_rx_buf[0]（Buffer 0）和 sbus_rx_buf[1]（Buffer 1）。
  • 开启了循环模式的直接内存写和双缓冲模式（DBM）。
  • 开启了 USART3 的全局接收，以及空闲中断（IDLE Interrupt）。
  1. 数据搬运阶段（纯硬件行为）
  • 当遥控器接收机顺着数据线发出这 18 字节包时，串口外设每收到 1 个字节，就命令 DMA 拿走。
  • DMA 像个自动传送带，默默地把这些字节顺序塞进当前的 Buffer 里，此时 CPU 在处理其他事情，没有任何察觉和中断开销。

  1. IDLE 触发阶段（一帧结束的标志） 一帧数据结束时（比如 18 个字节发完了），接下来的 1~2 毫秒内由于没有新数据线电平会一直保持空闲状态（高电平）。 这就触发了 USART3->SR & UART_FLAG_IDLE。此时，系统触发 USART3_IRQHandler 串口中断并进入处理流程。

  2. 中断处理与指针切换阶段（系统核心精华逻辑）

    1. 清除标志位：先调用 __HAL_UART_CLEAR_PEFLAG 清除空闲标志位。
    2. 判断当前数据在写哪个缓冲区（Buffer）：通过读取寄存器判断 (hdma_usart3_rx.Instance->CR & DMA_SxCR_CT) == RESET。如果是 RESET，说明 DMA 刚刚正在向 Buffer 0 里写东西。
    3. 计算并截断：
      • 暂停搬运：先强制执行 __HAL_DMA_DISABLE，暂停 DMA 对内存的写入，像拉下水闸一样。
      • 算算接了多少数据：用设定最大长度减去 DMA 内部还没写的倒计时寄存器，算出本次实际接收长度：this_time_rx_len = SBUS_RX_BUF_NUM - NDTR;
      • 重置计数器：强行将 NDTR 恢复成最大接收长度，随时准备接下一帧。
    4. 手动切换缓冲区：
      • 这行代码极为关键：hdma_usart3_rx.Instance->CR |= DMA_SxCR_CT;
      • 即使 Buffer 0 没满，代码也会强制告诉 DMA 硬件：你现在把枪口换对准另一个池子（Buffer 1）！
      • 执行完这步后，重启 DMA（__HAL_DMA_ENABLE），告诉它：继续干活吧。此时接下来不管传来什么数据，DMA都会老老实实将其放到崭新的 Buffer 1 开始的地方，这从根本上杜绝了上一帧丢字节引发后面全部发生错位对齐的灾难。
  1. 校验与封发阶段
  • 在这套切换缓冲区动作完成后，DMA 底层已经开始向新 Buffer 收下一个周期的数据去了。
  • 这时候中断函数里悠哉地去检查刚才算出的长度：if(this_time_rx_len == RC_FRAME_LENGTH) —— 收到的这包数据刚好等于 DBUS 协议规范的 18 字节吗？
  • 如果是精准无误的 18 字节数据帧，此时调用：rc_protocol->processByte(rc_protocol->sbus_rx_buf[0]);
  • 如果是残包（比 18 字节多或少），则被无情抛弃，等待下一帧重新覆盖。

3.3 Mahony姿态解算

参考文献：陀螺仪姿态解算+mahony滤波算法

Mahony算法是一种基于四元数的姿态解算方法，核心思想是利用陀螺仪的角速度积分预测姿态，同时用加速度计和磁力计的测量值修正预测误差。

算法流程：

  1. 初始化四元数：设定初始姿态为单位四元数 q = [1, 0, 0, 0]。

  2. 传感器归一化：将加速度计和磁力计测量值单位化，避免量纲影响。

  3. 计算参考方向：通过四元数旋转计算重力和地磁场在传感器坐标系下的期望方向。

  4. 误差计算：利用陀螺仪积分得到的姿态推算出的重力方向与加速度计测量得到的重力方向的向量叉乘得到姿态误差。

  5. PI反馈修正： 比例项（Kp）快速响应误差 积分项（Ki）抑制长期漂移

  6. 更新陀螺仪数据：将修正后的角速度用于新一轮四元数微分计算。

  7. 积分更新四元数（从世界坐标系到传感器坐标系的旋转）：

IMU启动瞬间利用加速度计和磁力计数据结合重力加速度推算出精确的初始姿态，即传感器在世界坐标系（人为定义的，一般是东北天）下的坐标。有了初始姿态，之后进入预测-更新循环，即上述过程。

由于RM中只需要yaw和pitch两个轴即可，姿态解算中的绝对参考有加速度计确定picth和roll。
